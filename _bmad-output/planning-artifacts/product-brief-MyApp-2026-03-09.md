# Product Brief: HuoTong（货通）

## Executive Summary

HuoTong（货通） 是一款专为小型个体商户（厨具/日杂）设计的轻量级业务管理工具。目标用户是一个仅有 4 人的家庭式商户团队，当前完全依赖纸质单据和人脑记忆进行日常经营管理。HuoTong（货通） 旨在用最简单的数字化方式解决他们在单据管理、商品查价、客户和供应商信息管理、库存追踪和账款对账等环节的效率和遗漏问题。由家人兼开发者主导开发，确保产品完全贴合实际使用场景。

本轮产品方向调整为：**以 Android 客户端作为唯一主交付形态**。现有 Web/H5 方案保留为已验证的业务原型和迁移基础，但不再作为继续演进的正式产品入口。这样做的核心原因不是功能变化，而是为了绕开公网网页入口、域名与备案、默认托管域名风险提示等交付摩擦，让家人可以像打开普通 App 一样直接使用系统。

---

## 核心愿景

### 问题陈述

小型个体商户（经营厨具、日杂用品）的日常运营完全依赖纸质单据和人脑记忆。商品价格靠记忆和翻找上货单，客户和供应商信息散落在手机通讯录、纸条和脑子里，进出货记录靠手写单据，应收应付账款靠定期手动整理。这种管理方式导致两个核心问题：**浪费大量时间**在翻找、整理、回忆上，以及**容易遗漏**导致漏账、错价、断货等损失。

### 问题影响

- **时间浪费**：每次查价、对账、盘库存都需要翻找纸质记录或凭记忆，占用大量本应用于经营的时间
- **信息遗漏**：纸质记录易丢失、不可搜索，人脑记忆不可靠，导致漏收账款、错误报价、库存断档
- **人员依赖**：核心信息集中在老板一个人脑中，其他人无法独立处理业务，团队效率受限
- **决策盲区**：缺乏数据积累，无法了解什么好卖、什么滞销、哪个供应商性价比高

### 为什么现有方案不够

市面上的进销存软件（如管家婆、秦丝等）主要面向规模更大的商户设计，存在以下问题：
- **过于复杂**：功能繁多，4人小商户用不到80%的功能，学习成本高
- **不够贴合**：通用方案无法针对具体商户的操作习惯和业务流程做定制
- **成本门槛**：按年收费的模式对小商户来说性价比不高
- **上手困难**：忙碌的小商户没有时间和耐心学习复杂软件

### 解决方案

HuoTong（货通） 采用"极简定制"策略——以最轻量的方式覆盖核心业务环节。Android 首版聚焦 6 个核心功能模块：

1. **数字出货/进货单**：替代纸质单据，自动关联商品、客户、供应商，串联所有数据
2. **商品价格库**：随时可查可改，新人也能即刻上手报价
3. **应收应付追踪**：实时记录每笔账的状态，一眼看清谁还没付款
4. **客户信息管理**：统一客户档案，带交易历史记忆
5. **供应商信息管理**：集中供应商资料，进货时快速查找对比
6. **库存管理**：出入货自动增减，实时掌握库存状态

### 关键差异化

- **极致贴合**：由家人兼开发者主导，零距离观察真实场景，每个操作流程为这个团队量身定做
- **极致简单**：只做最必要的功能，每个操作追求"3秒内完成"，比纸笔更快才有人用
- **数据联动**：不是5个独立电子表格，而是以单据为纽带串联所有模块——出一笔货，库存自动减、应收自动加、客户记录自动更新
- **持续迭代**：开发者就在身边，随时根据使用反馈调整优化
- **交付阻力更低**：从"打开网页"转为"点击 Android 图标"，尽量减少浏览器、域名、备案和网络入口带来的额外摩擦

## 目标用户

### 主要用户

**用户画像：一家人（4人家庭商户团队）**

**人物 1：父母（核心经营者）**
- **角色**：商户的实际经营主力，进货、卖货、对账全程参与
- **数字化能力**：会用微信，能发消息、看朋友圈、微信支付的水平
- **痛点**：多年经营积累的商品价格、客户关系全在脑子里，翻纸质单据费时费力，怕漏账
- **核心需求**：操作必须像发微信一样简单，不能有复杂的表单和流程
- **成功标准**：查个价格比翻本子快、记笔账比写单据省事，他们才会用

**人物 2：女儿（数字化桥梁）**
- **角色**：会用电脑，是家里数字化能力最强的成员
- **数字化能力**：会用电脑和手机，可以承担初始数据录入、系统配置等较复杂的操作
- **特殊价值**：可以负责前期商品信息批量录入、帮家人学习使用、处理需要电脑操作的任务
- **核心需求**：希望帮家里减轻负担，工具能让全家人都用得起来
- **成功标准**：不需要她时刻在场，家人也能独立完成日常操作

### 本轮交付形态调整

- **主交付形态**：Android 客户端（Capacitor 增量改造，复用现有 Vue/Vite 业务代码）
- **非目标形态**：Web/H5/PWA 不再作为正式入口继续投入
- **首版原则**：先保证核心开单、查价、记账流程稳定可交付，再逐步恢复拍照识别、导出、实时同步等增强能力
- **安装方式**：默认接受家庭内部手工安装 APK，不以应用商店首发为前提

### 次要用户

**开发者（家人兼开发者）**
- **角色**：产品的设计、开发和维护者
- **特殊价值**：零距离观察真实使用场景，可以随时收集反馈和迭代
- **核心需求**：需要了解使用情况以便持续优化（如简单的使用日志或反馈入口）

### 用户旅程

**第一阶段：冷启动（女儿主导）**
- 女儿帮忙录入现有商品信息、常用客户和供应商资料
- 搭建基础数据底座，让系统"有东西可查"

**第二阶段：试用过渡（双轨并行）**
- 日常出货/进货先用系统录一遍，同时保留纸质单据作为备份
- 家人逐渐习惯"先查系统"而不是"先翻本子"
- "啊哈时刻"：第一次通过系统3秒查到一个价格，或者发现有笔账差点忘了收

**第三阶段：日常依赖（全员使用）**
- 纸质单据逐步退出，系统成为日常经营的核心工具
- 每个人都能独立完成查价、开单、记账等基本操作
- 数据开始积累，为二期功能（补货建议、畅销分析等）打下基础

### 关键设计约束

- **界面**：大字体、大按钮、高对比度，适合各年龄段
- **操作**：手机端为主，核心操作不超过3步点击
- **学习**：零学习成本——看到就会用，像微信一样直觉化
- **容错**：允许修改和撤销，不怕录错
- **交付**：首次安装和更新步骤必须足够简单，能够由家人在指引下完成
- **网络**：首版默认依赖联网，但遇到弱网/断网时必须给出清晰反馈，不允许静默卡死

## 成功指标

### 用户成功标准

- **核心标准**：家人愿意主动使用，而不是被催着用
- **替代标准**：日常经营中逐步不再依赖纸质单据
- **体验标准**：常用操作（查价、开单、记账）比翻纸质本子更快
- **交付标准**：家人能顺利安装 App，并把它当成日常打开的主要入口

### 业务目标

本产品为家庭自用工具，不涉及商业化目标。成功的唯一衡量是：**用起来比不用好**。

### 关键指标

- 不漏账
- 不错价
- 省时间
- 安装后能稳定登录并恢复会话
- 弱网下主流程失败时有明确提示和重试路径

## MVP 范围

### 核心功能

**MVP 包含 6 个功能模块，以"单据"为数据纽带串联：**

**1. 数字出货/进货单**
- 手机端快速创建出货单和进货单
- 选择客户/供应商 → 添加商品 → 确认金额 → 完成
- 出货自动触发：库存减少 + 应收增加 + 客户记录更新
- 进货自动触发：库存增加 + 应付增加 + 供应商记录更新

**2. 商品价格库**
- 商品名称、规格、售价、进价录入和查询
- 支持模糊搜索（输入关键字即可找到商品）
- Android 首版优先保证单条录入与查询顺畅，批量录入不作为本轮继续演进重点

**3. 应收应付追踪**
- 每笔单据自动生成应收/应付记录
- 状态标记：未付 → 部分付 → 已付
- 按客户/供应商查看汇总欠款

**4. 客户信息管理**
- 客户姓名、电话、地址等基本信息
- 关联该客户的历史出货单和应收账款

**5. 供应商信息管理**
- 供应商名称、联系人、电话、主营品类
- 关联该供应商的历史进货单和应付账款

**6. 库存管理**
- 商品库存数量随出入货自动增减
- 库存查询：当前有多少、最近出了多少

### MVP 不包含的功能

以下功能明确不在第一版范围内：

| 功能 | 推迟原因 |
|------|---------|
| 智能补货建议 | 需要先积累足够库存数据 |
| 畅销/滞销分析 | 需要先积累足够销售数据 |
| 商品目录展示 | 非核心经营需求 |
| 出货状态追踪 | MVP 阶段用简单备注即可 |
| 多价格体系 | 第一版先用单一售价，后续扩展 |
| 历史价格追踪 | 第一版先解决"能查到价格"的问题 |
| 单据模板/快速复制 | 使用频率需要验证后再做 |
| App 应用商店发布 | 首版采用家庭内部 APK 分发即可 |
| Realtime 实时同步 | 首版以稳定为先，必要时默认关闭 |
| 拍照识别原生化 | 首版优先保证手工录单闭环，后续再接原生相机能力 |
| 导出文件到手机 | 需要原生文件能力，后续独立规划 |

### MVP 成功标准

- 家人能独立完成开单、查价、记账操作
- 常用操作比翻纸质本子更快
- 数据不丢失、不出错
- Android 安装与更新流程可被家人理解并完成
- 首版即使关闭拍照识别和实时同步，核心业务仍然可用

### 未来愿景

第一版跑通后，根据实际使用反馈逐步添加二期功能。优先级由实际使用中暴露的需求决定，而非提前规划。
