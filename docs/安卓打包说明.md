# 安卓打包说明

本文档说明如何在当前仓库中为 `HuoTong` 打出 Android 调试包和发布包，并指出和本项目现状相关的关键配置。

配套文档：

- `docs/android-capacitor-init.md`
- `docs/android-app-identity-signing.md`
- `docs/android-apk-distribution.md`

## 1. 当前项目打包信息

- 前端目录：`huotong-app/`
- Android 工程目录：`huotong-app/android/`
- 包名 / applicationId：`com.huotong.app`
- 当前版本号：
  - `versionName = 1.0.1`
  - `versionCode = 2`
- Web 资源目录：`dist`
- 最低 Android 版本：`minSdkVersion = 24`
- 推荐/要求 JDK：`21+`

常用脚本位于 `huotong-app/package.json`：

```bash
npm run android:sync
npm run android:build
npm run android:build:release
npm run android:open
npm run android:run
```

## 2. 打包前准备

在 `huotong-app/` 下操作。

### 必备环境

- Node.js 20+
- JDK 21 或更高版本
- Android Studio
- Android SDK / Gradle 环境可正常使用

### 安装依赖

```bash
cd huotong-app
npm install
```

### 检查 Java 版本

```bash
java -version
```

本项目的 `android:build`、`android:build:release`、`android:run` 会先执行 `scripts/ensure-android-java.mjs`。  
如果当前 Java 主版本低于 21，脚本会直接失败，并提示你切换 `JAVA_HOME`。

### 推荐：用 direnv 在项目内固定 JDK 21（不影响全局）

如果你本机还有 Java 8/11 等旧版本，建议只在 `huotong-app/` 目录启用 JDK 21。

1. 安装并启用 `direnv`（只需做一次）：

```bash
brew install direnv
echo 'eval "$(direnv hook bash)"' >> ~/.bash_profile
source ~/.bash_profile
```

2. 在 `huotong-app/.envrc` 写入固定 JDK 21 路径：

```bash
export JAVA_HOME="/opt/homebrew/opt/openjdk@21/libexec/openjdk.jdk/Contents/Home"
export PATH="$JAVA_HOME/bin:$PATH"
```

3. 授权并验证：

```bash
cd huotong-app
direnv allow
direnv exec . java -version
```

预期输出应为 `openjdk version "21.x"`。  
后续在该目录中运行 `npm run android:build` / `npm run android:build:release` 即可自动使用 JDK 21。

### 检查前端环境变量

Android 包内加载的是本地前端资源，但前端资源本身仍然依赖 Supabase 等环境变量。  
打包前至少确认 `huotong-app/.env` 或 `.env.local` 中需要的 `VITE_*` 配置已经准备好。

常见关键项：

```env
VITE_SUPABASE_URL=...
VITE_SUPABASE_ANON_KEY=...
```

如果你在打包 Android 首版，还应确认是否需要这些开关：

```env
VITE_ANDROID_FIRST_RELEASE_MODE=true
VITE_REALTIME_ENABLED=false
VITE_RECEIPT_RECOGNITION_ENABLED=false
VITE_WEB_EXPORT_DOWNLOAD_ENABLED=false
VITE_DESKTOP_BATCH_IMPORT_ENABLED=false
```

## 3. 调试包打包

调试包适合本机安装、真机联调、回归验证，不要求配置正式签名。

```bash
cd huotong-app
npm run android:build
```

这个命令会自动执行：

1. 检查 JDK 是否为 21+
2. 运行 `npm run build`
3. 运行 `npx cap sync android`
4. 执行 `./android/gradlew -p android assembleDebug`

### 调试包输出位置

通常输出在：

```bash
huotong-app/android/app/build/outputs/apk/debug/app-debug.apk
```

### 常见用途

- 首次验证 Android 工程是否还能正常构建
- 真机安装做功能验收
- 回归测试返回键、前后台恢复、弱网提示、覆盖安装前检查

## 4. 发布包打包

发布包用于分发给家人安装、覆盖升级或留档。  
发布包要求签名配置完整。

```bash
cd huotong-app
npm run android:build:release
```

这个命令会自动执行：

1. 检查 JDK 是否为 21+
2. 检查签名配置是否完整
3. 运行 `npm run build`
4. 运行 `npx cap sync android`
5. 执行 `./android/gradlew -p android assembleRelease`

### 发布包输出位置

通常输出在：

```bash
huotong-app/android/app/build/outputs/apk/release/app-release.apk
```

## 5. 发布签名配置

### 第一步：准备 `keystore.properties`

仓库中已有模板文件：

```bash
huotong-app/android/keystore.properties.example
```

复制为：

```bash
cp huotong-app/android/keystore.properties.example huotong-app/android/keystore.properties
```

内容格式如下：

```properties
storeFile=../keystore/huotong-release.jks
storePassword=你的密码
keyAlias=huotong-release
keyPassword=你的密码
```

注意：

- `keystore.properties` 已被 git 忽略，不要提交
- `storeFile` 是相对于 `huotong-app/android/` 的路径
- `scripts/ensure-android-signing.mjs` 会检查这 4 个字段是否齐全，并检查 keystore 文件是否真实存在

### 第二步：准备 keystore

如果是首次发版，还没有 keystore，可在 `huotong-app/` 下执行：

```bash
mkdir -p keystore
keytool -genkeypair \
  -v \
  -keystore ./keystore/huotong-release.jks \
  -alias huotong-release \
  -keyalg RSA \
  -keysize 2048 \
  -validity 3650
```

### 第三步：备份 keystore

- 至少保留 2 份离线备份
- 不要通过聊天工具明文发送
- 不要修改 alias、keystore 或密码后忘记同步记录

一旦线上已有安装用户，后续覆盖升级必须继续使用同一套签名。

## 6. 版本号修改

当前版本定义在：

```bash
huotong-app/android/app/build.gradle
```

当前值是：

```gradle
versionCode 2
versionName "1.0.1"
```

发新版本前，至少执行：

1. 修改 `versionName`
2. 把 `versionCode` 严格加 1
3. 再打 release 包

示例：

```gradle
versionCode 3
versionName "1.0.2"
```

注意：

- `versionCode` 不递增会导致覆盖安装失败
- 已发布后不要随意改 `applicationId`，否则新包无法覆盖旧包

## 7. 推荐打包流程

每次准备发 Android 包时，建议按下面顺序做：

1. 进入 `huotong-app/`
2. 确认 `.env*` 中本次需要的前端配置和功能开关
3. 确认 `android/app/build.gradle` 中的 `versionName` / `versionCode`
4. 若是发布包，确认 `android/keystore.properties` 和 `keystore/*.jks` 正确
5. 运行 `npm run test:run`
6. 运行 `npm run android:build` 做一次调试包验证
7. 运行 `npm run android:build:release` 产出发布包
8. 在真机上执行一次覆盖安装验证

## 8. 覆盖安装验证

发布给家人前，至少做一次真机覆盖安装：

1. 先安装旧版 APK
2. 打开 App，确认能进入登录页/首页
3. 不卸载旧版，直接安装新版 APK
4. 确认安装成功，没有提示签名冲突或版本回退
5. 打开 App，确认能正常启动
6. 确认登录态、首页和一个关键业务页可正常访问
7. 到“更多”页确认当前版本或更新说明与本次发版一致

如果覆盖安装失败，优先检查：

- `applicationId` 是否被改动
- 新旧 APK 是否使用同一 keystore
- `versionCode` 是否严格递增

## 10. 常见问题排查

### `Android build requires JDK 21+`

原因：当前 `java -version` 小于 21。  
处理：安装 JDK 21+，并把 `JAVA_HOME` 指向正确版本。

### `Missing android/keystore.properties`

原因：执行 release 打包时没有签名配置。  
处理：从 `android/keystore.properties.example` 复制并填写真实值。

### `Keystore file not found`

原因：`keystore.properties` 里的 `storeFile` 路径不对，或文件不存在。  
处理：检查相对路径是否以 `huotong-app/android/` 为基准。

### 覆盖安装失败

优先检查：

- 是否改了 `applicationId`
- 是否更换了签名证书
- 是否忘了递增 `versionCode`

### 安装后页面还是旧内容

原因通常是前端资源没重新构建或没同步。  
处理：重新执行：

```bash
cd huotong-app
npm run android:sync
```

如果仍有问题，再重新执行完整打包命令。

### 真机登录提示“网络异常”，模拟器却正常

常见原因：打包时使用了真机网络不可达的 Supabase 域名（例如某些网络环境下 `supabase.opentrust.net` 路由超时），导致真机登录失败，而模拟器因网络出口差异仍可用。  
处理建议：

1. 先在真机上验证 Supabase 健康检查接口可达（通过 `adb shell curl -I`）。
2. 若自定义域名超时，优先切回官方项目域名（如 `*.supabase.co`）并保证与 `VITE_SUPABASE_ANON_KEY` 成对。
3. 重新执行 `npm run android:build` 或 `npm run android:build:release` 生成新包后覆盖安装。
4. 覆盖安装前确认签名一致；若 `debug` 包签名不一致，改用同签名的 `release` 包安装。
